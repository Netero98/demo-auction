# Зачем вам этот форк ?

Данный форк добавляет к оригиналу лишь этот документ. Этот документ решает серьезный недостаток основного курса - он является лишь логом действий автора, что хорошо для объяснения, но не даёт понимания, как использовать актуальную версию проекта в случае, если вы захотите начать пилить, например, стартап на базе данного проекта (который весьма хорошо продуман для этой цели). Чтобы начать делать свой проект, вам придётся просмотреть весь курс и дойти до актуальной версии кодовой базы. Тут лежит выжимка из курса, которая поможет вам без просмотра более 100 часов основного материала сайта Deworker Pro уже сегодня запустить полноценный проект на основе самой актуальной версии кодовой базы. Информация дана в виде неупорядоченных топиков.

## Как поднять проект локально для разработки?
make init. Это главная команда при локальной разработке, суть её в том, чтобы полностью с нуля собрать и поднять проект, обновив всё что только можно и удалив перед этим всё старое включая базу данных. С этой команды надо начинать рабочий день. Для её выполнения нужен только установленный Docker на локальной машине.

## Деволтный сценарий создания новой фичи по методологии TDD, BDD
- Прогоняем make check. Убеждаемся, что ВСЕ старые тесты рабочие 
- Пишем свои е2е и другие тесты.
- e2e тесты оформляем в виде фич в директории cucumber/features. 
- Прогоняем make check ещё раз, убеждаемся, что все новые тесты упали (красная стадия)
- Пишем минимально рабочий код чтоб тесты не упали 
- Если появились новые требования, повтоярем цикл. Если код говно, рефакторим и прогоняем тесты. Если требований больше нет, идём дальше по воркфлоу (ревью, деплой и т.д.)

## Каким образом устроено документирование проекта?
Автотесты - это и есть документация. От обычной документации отказались т.к. все всегда на неё забивают,
она со временем теряет связь с реальным состоянием проекта.
Если тесты добавлены в пайплан, на них невозможно забить. Тесты должны быть понятны как
программисту, так и аналитику. Аналитик ставит фича задачи разработчику в виде e2e фич, описанных понятным человеку
языком и пригодных для дальнейшей автоматизации.

## Все мониторинговые задачи, которые надо делать с периодичностью
- Обновление всех composer, js зависимостей проекта
- Убедиться, что никто не выпилил какие-то проверки из пайплайна, что отрабатывают все тесты и т.п. чтобы исключить
вероятность того, что мы живём в иллюзии безопасности
- Отрабатывать самый худший возможный сценарий: нас взломали, базы данных и все дампы зашифровали, сервера сгорели и т.п.
Нужно иметь задокументированный сценарий, по которому проект при любом раскладе подымет даже джун, а не какой-то супер разработчик,
который уволился 3 года назад.
- Нужно проводить аудит безопасности той глубины, которая необходима для вашего типа бизнеса.
- Нужно актуализировать документацию, чтобы даже джун смог по ней минимально поддерживать проект. Цель - у новичка 0 вопросов после прочтения доки. Даже если дока превратится в войну и мир, это всё равно дешевле альтернатив.

## Как работают е2е тесты?
Вместо тяжеловесного селениума используется Puppeteer. Тесты прогоняются без UI, т.к.
действие происходит внутри cucumber контейнера с использованием chromium, который позволяет прогонять тесты без UI.
